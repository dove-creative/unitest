# UniTest 테스트 작성 워크플로

이 문서는 에이전트가 UniTest 기반 테스트를 작성하거나 보강할 때 따르는 작업 흐름이다.

---

## 목차

1. 모드 선택
2. 테스트 문서 폴더 준비
3. 상태-동작 표 작성
4. 테스트 계획 작성
5. 테스트 코드 작성
6. 빌드와 실행
7. 결과 기록과 통합 테스트

---

## 1. 모드 선택

테스트 작성 작업을 시작할 때 설계자가 지정한 모드를 먼저 확인한다. 모드가 지정되지 않았다면 안전 모드를 기본값으로 사용한다.

| 모드 | 실패 또는 설계 오류 대응 |
| --- | --- |
| 안전 모드 | 테스트 실패 시 원인과 필요한 수정안을 설계자에게 보고하고 대기한다. |
| 자동 모드 | 테스트 실패 시 테스트 코드 또는 대상 코드를 직접 수정한다. API 변경이 필요한 중대한 설계 오류일 때만 설계자에게 보고하고 대기한다. |
| 위임 모드 | API 변경이 필요한 경우에도 에이전트가 스스로 코드와 문서를 수정하고, 작업이 끝난 뒤 변경 내용과 결과를 보고한다. |

모드는 실패 대응 범위를 정하는 기준이다. 상태-동작 표 작성, 테스트 계획, 테스트 코드 작성, 실행 결과 기록의 기본 흐름은 모든 모드에서 동일하게 유지한다.

---

## 2. 테스트 문서 폴더 준비

위키의 해당 모듈 폴더에 `UniTests` 폴더를 생성한다. 이미 해당 모듈에 `Tests` 폴더가 존재한다면, 그 폴더 아래에서 테스트 문서를 이어서 관리한다.

테스트 문서는 기본적으로 다음 파일로 나눈다.

| 파일 | 역할 |
| --- | --- |
| `01-Tables.md` | 모듈별 상태-동작 표 |
| `02-Plans.md` | 상태-동작 표를 코드로 옮기는 테스트 계획 |
| `03-Results.md` | 실행 중 확인한 주요 이슈와 결과 기록 |

---

## 3. 상태-동작 표 작성

`01-Tables.md` 파일에 각 모듈 또는 유닛의 상태-동작 표를 작성한다. 기존 `01-Tables.md` 파일이 있다면, 현재 구현과 공개 API를 기준으로 내용을 검토하고 보강한다.

상태-동작 표의 형식과 의미는 [UniTest 상태-동작 지침](../Wiki.ko/00-Unit-Test-Guideline.md)을 따른다.

---

## 4. 테스트 계획 작성

상태-동작 표를 바탕으로 테스트 코드를 작성하거나 수정하기 전에 `02-Plans.md` 파일을 만든다. `02-Plans.md`는 표의 어떤 칸을 어떤 테스트 파일과 테스트 함수로 옮길지 정리하는 계획 문서이다.

표의 모든 테스트 칸이 200개 이하라면 모든 테스트를 작성한다. 200개를 초과한다면 안전 모드와 자동 모드에서는 예상되는 테스트 개수를 설계자에게 보고하고 잠시 대기한다. 위임 모드에서는 테스트 범위를 세부 항목으로 나누고, 각 항목을 별도 폴더의 독립 테스트로 분리해 수행한다.

테스트 계획은 다음 기준으로 작성한다.

- 테스트는 가능하면 객체별로 파일을 분리한다.
- UniTest 프레임워크를 사용할 경우, 테스트 단계와 파일 구성을 어떻게 작성할지 개괄적인 계획을 적는다.
- UniTest 프레임워크를 사용하지 않을 경우, 표에 기반하여 실행할 테스트들의 개괄적인 계획과 목록을 적는다.
- 단위 테스트와 통합 테스트를 구분하고, 통합 테스트는 대표 유스케이스와 실패 시나리오를 중심으로 계획한다.

---

## 5. 테스트 코드 작성

테스트 계획에 따라 테스트 파일을 작성하거나 수정한다. 테스트 코드는 해당 모듈 또는 유닛 폴더 아래의 `UniTests` 폴더에 둔다. 이미 해당 모듈에 `Tests` 폴더가 존재한다면, 그 폴더 아래에서 `UniTests` 폴더를 생성하여 사용한다.

도메인별 POCO 테스트를 Unity 밖에서 실행해야 하는 경우에도 테스트 프로젝트는 UniTest 패키지 안에 추가하지 않고, 해당 도메인 폴더가 자기 external test project를 소유한다.

테스트 어셈블리는 `모듈명.Tests.UniTest.asmdef` 파일로 분리하고, Unity Editor의 Test Runner에서 감지될 수 있도록 설정한다.

테스트 대상 프레임워크가 POCO라면 가능하면 Unity 프레임워크를 사용하지 않도록 테스트를 구성한다. Unity API, Scene, AssetDatabase, PlayMode, Editor API가 필요하지 않은 테스트는 [External NUnit Executor 워크플로](02-External-NUnit-Executor-Workflow.md)를 통해 `dotnet test` 빌드와 실행 대상이 될 수 있도록 구성한다. Unity 실행 환경이 필요한 테스트는 억지로 POCO 테스트로 바꾸지 않고 Unity Test Runner 또는 Unity batchmode 검증으로 분리한다.

각 테스트 함수에는 짧은 주석을 작성하여, `01-Tables.md`와 `02-Plans.md`의 어떤 단계 또는 칸에 해당하는 테스트인지 알아볼 수 있게 한다.

테스트를 작성한 뒤에는 완성된 테스트와 계획 문서를 대조한다.

- 계획에서 누락된 테스트가 있다면 테스트를 추가한다.
- 구현 중 추가되거나 변경된 테스트가 있다면 `02-Plans.md`에 반영한다.
- 테스트 코드가 현재 표와 맞지 않는다면 먼저 표 또는 계획의 기준을 갱신한 뒤 코드를 맞춘다.

---

## 6. 빌드와 실행

테스트 코드를 빌드하고 실행한다. 자동 빌드가 불가능하거나 현재 환경에서 Unity Test Runner 실행이 필요한 경우에는 설계자에게 테스트 실행을 요청한다.

실행 경로는 다음 기준으로 선택한다.

| 대상                              | 실행 경로                                                                                              |
| ------------------------------- | -------------------------------------------------------------------------------------------------- |
| Unity API 또는 Editor 환경이 필요한 테스트 | Unity Test Runner 또는 Unity batchmode                                                               |
| UniTest 기반 POCO 테스트             | NuGet `NUnit` 3.x를 복원해 실행하는 `dotnet test` 기반 [External NUnit Executor 워크플로](02-External-NUnit-Executor-Workflow.md) |

테스트 실행 중 실패가 발생하면, 실패한 테스트가 표의 어떤 칸과 계획의 어떤 항목에 대응하는지 먼저 확인한다.

실패 이후의 보고, 수정, 대기 여부는 1번에서 선택한 모드를 따른다.

---

## 7. 결과 기록과 통합 테스트

실행 도중 발생한 주요 이슈를 기록할 필요가 있다면 `03-Results.md` 파일에 기록한다.

기록 대상은 다음과 같다.

- 빌드 또는 실행 명령
- 성공과 실패 결과
- Unity 환경 또는 CLI 환경의 제약
- 실패 원인과 후속 조치
- 표, 계획, 코드 사이의 정합성 보강 내용

모듈 또는 유닛 테스트가 완료되면 대표 유스케이스와 실패 시나리오를 포함해 통합 테스트를 수행한다. 통합 테스트의 실행, 보고, 대기 방식도 1번에서 선택한 모드를 따르며, 필요한 결과는 `03-Results.md`에 함께 기록한다.
