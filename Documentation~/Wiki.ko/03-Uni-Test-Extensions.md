## 1. Failure Stop/Re-Execute

Uni Test는 가능한 모든 테스트 케이스를 전부 실행하므로, 시스템 규모가 클수록 전체 수행 시간이 길어질 수 있다. 이때 테스트 도중 실패가 발생하면, 개발자는 즉시 코드를 수정한 뒤 실패한 테스트만 다시 실행하여 수정 결과를 확인하고자 할 수 있다. 이를 지원하기 위해 Uni Test는 아래의 Failure Stop과 Re-Execute 두 기능을 제공한다. 두 기능은 연계 사용이 가능하며, Failure Stop으로 얻은 실패 목록을 곧바로 Re-Execute의 입력으로 넘겨 빠른 검증 사이클을 수행할 수 있다.

1. **Failure Stop**: 하나라도 실패하면 즉시 나머지 테스트 실행을 중단하고, 실패한 테스트 목록을 반환한다.
2. **Re-Execute**: 지정한 일련의 테스트 집합만 다시 실행한다.

두 기능은 아래와 같이 구현한다.

1. **Failure Stop**: Test Executor가 예외 발생/Assert 실패 등을 감지하면 Project의 플래그를 `true`로 설정한다. Project는 각 실행 루프마다 해당 플래그를 확인하여 변경이 감지되면 즉시 전체 실행을 중단한다.
2. **Re-Execute**: 입력받은 일련의 테스트 ID를 기준으로 필터링 목록을 구성한다. 이후 새 Model 인스턴스를 생성한 뒤 TestDesigner에게 테스트 생성을 위임하고, 생성된 테스트 중 필터 목록에 해당하는 테스트만 실행 큐에 남기는 방식으로 테스트를 재실행한다.

## 2. Node의 재실행과 디버그

테스트 실패 시, 개발자는 Uni Test의 XML 리포트를 통해 주요 원인을 확인할 수 있다. 그러나, 일부 경우 리포트만으로는 확인이 어려운 세부 실행 흐름이나 상태 정보를 직접 점검해야 하는 경우도 발생할 수 있다. 이때 개발자는 Uni Test가 반환한 Node 목록에서 원하는 노드를 선택해 IDE에서 재실행함으로써 중단점, 변수 감시, 스택 추적 등을 이용해 문제 지점을 면밀히 분석할 수 있다.

노드 재실행 절차는 다음과 같다.

1. `node.DetachAndRestore()`를 호출하여 원본 Node 그래프를 변경하지 않는 분리된 디버그용 Node를 생성한다.
2. 반환된 Node에 대해 `restored.Execute()`를 호출하여 동일한 실행 이력을 재현한 상태에서 테스트를 다시 실행한다.

이 과정을 통해 실패 상황을 재현하고, 코드 레벨에서 원인을 심층적으로 검증할 수 있다.

## 3. 연속 테스트

Uni Test는 가능한 모든 테스트 케이스를 실행하기 때문에, 테스트 단계가 많아질수록 전체 수행 시간은 $O(n^2)$ 수준으로 증가한다. 이로 인해 테스트 단계가 많아지면 실행 비용이 급격히 증가하는 문제가 발생한다.

이를 보완하기 위해 Uni Test는 가능한 모든 경로를 전부 실행하는 대신, 성공한 테스트 중 연속 가능한 하나의 경로를 임의로 선택하여 실행하는 옵션을 제공한다. 이 옵션을 사용하면 전체 소요 시간을 $O(n)$ 수준으로 줄일 수 있어, 많은 테스트를 통해 객체의 안정성을 검증해야 하는 상황에 효과적으로 대응할 수 있다.
