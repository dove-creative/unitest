## 1. Uni Test 소개

Uni Test는 다양한 상태 전환과 기능을 가진 테스트들을 자동화된 방식으로 테스트하기 위해 설계된 도구이다.

이 도구는 상태 테이블 기반의 테스트 설계를 통해 가능한 모든 테스트 케이스를 자동으로 생성하고 실행함으로써, 복잡한 테스트에서도 높은 신뢰성을 확보할 수 있도록 돕는다.

테스트는 AAA (Arrange-Act-Assert) 패턴을 기반으로 구성되며, 객체의 상태 전환과 연속 동작을 중심으로 테스트를 수행한다.

| 분류  | 내용                            |
| --- | ----------------------------- |
| 개요  | 다양한 상태와 인터페이스를 가지는 테스트의 자동 수행 |
| 목적  | 시스템의 신뢰성 또는 검증                |
| 구현  | 상태 테이블 기반의 테스트 자동 설계 및 실행     |

Uni Test는 다음과 같은 특징을 가진다.

AAA 패턴 기반의 테스트 구성
- Arrange: 테스트에 필요한 객체와 상태 설정
- Act: 테스트 동작 실행
- Assert: 결과 검증

상태 기반 테스트 지원
- 상태 테이블을 통해 객체의 다양한 상태와 동작을 정의
- 상태 전환을 포함한 연속적인 테스트 설계 및 실행

테스트 설계 자동화
- 가능한 모든 테스트 케이스를 자동 생성
- 테스트 종료 후 다음 테스트로 이어지는 자동 전환

## 2. 하나의 상태를 가지는 테스트에 대한 테스트
### 2-1. 구성 요소

#### 2-1.1. Model

Model은 테스트 대상 객체와 그 예상 상태, 실행 데이터를 포함한 메타 정보를 함께 보유하는 단위 객체이다.

Model은 다음과 같은 요소로 구성된다.

<img src="../Resources/01-20260420213325.png" alt="UniTest model structure diagram" width="334">

구성 요소
- Subject: 실제 테스트를 수행할 대상 객체
- Mock: 테스트 대상이 되는 상태를 묘사하는 객체
- Metadata: 실행 횟수, 지연 시간 등 테스트에 필요한 실행 정보

#### 2-1.2. Lab

Lab은 테스트에 사용되는 세 개의 함수(Arrange, Act, Assert)를 포함하며, 외부에서 Model을 입력받아 순차적으로 테스트를 실행하는 주체이다.

Lab은 다음과 같은 요소로 구성된다.

<img src="../Resources/01-20260420213332.png" alt="UniTest lab structure diagram" width="902">

구성 요소
- Arranger: 테스트 실행 전 Model을 설정하고, Mock을 Subject의 예상 상태로 설정
- Actor: 실제 테스트 동작 실행
- Asserter: Subject가 예상대로 작동하는지 검증
함수
- Execute: 입력 Model에 대해 Arrange → Act → Assert 순서로 테스트 실행

#### 2-1.3. Node

Node는 하나의 테스트 실행 단계를 표현하며, Model과 Lab의 쌍으로 구성된다. 또한, 연속되는 테스트 흐름 상에서 이전/이후의 Node를 참조하여 테스트 흐름을 추적할 수 있도록 돕는다.

Node는 다음과 같은 요소로 구성된다.

<img src="../Resources/01-20260420213333.png" alt="UniTest node structure diagram" width="476">

구성 요소
- Model: 현재 단계에서 사용될 Model
- Lab: 현재 단계에서 실행될 Lab
- Before: 이전 단계에 해당하는 Node
- Afters: 이후 단계에 해당하는 Node들의 집합

함수
- Execute: 현재 Node의 Model을 기반으로 Lab.Execute 실행
- Append: 새로운 Lab을 입력받아 후속 Node 생성

Append가 호출되면, Node는 새로운 Model을 만든 후 지금까지 실행된 모든 Lab을 순차적으로 다시 실행하여 현재 Model의 복사본을 만든다. 이렇게 생성된 Model은 후속 Node에 전달되어, 각 실험 단계 간의 상태가 서로 간섭하지 않도록 한다.

### 2-2. 테스트 실행

#### 2-2.1. 개요

UniTest는 Node를 단위로 하여 테스트를 실행한다. 이때 테스트는 다음과 같은 주기로 진행된다.
1. Design: 현재 Node.Model에서 실행 가능한 Labs를 생성한다. 이들을 바탕으로 새로운 후속 Nodes 생성
2. Execute: Design 단계에서 생성된 각 Node에 대해 Node.Execute를 수행하여 테스트 실행
3. 테스트 자동 종료: Node.Model.Metadata를 바탕으로, 계속 테스트를 실행할 수 있는 Node는 Design 단계로 이동, 그렇지 않은 Node는 Finished Node로 처리

#### 2-2.2. Project

UniTest에서 Node를 통해 테스트를 실행하는 객체를 Project라고 한다.
Project는 다음과 같은 요소로 구성된다.

<img src="../Resources/01-20260420213335.png" alt="UniTest project structure diagram" width="903">

구성 요소
- Root Node: 테스트의 시작점이 되는 Node로 최초 Idle Node에 해당
- Idle Nodes: 테스트 주기의 첫 번째 단계(Design)를 대기 중인 Node들의 집합
- Prepared Nodes: 테스트 주기의 두 번째 단계(Execute)를 대기 중인 Node들의 집합

함수
- Execute: 입력받아 전체 테스트 과정 실행
- Create Labs (가상 메서드): Design 과정 중 주어진 Node.Model에 대해 실행 가능한 Labs 생성

## 3. 다수의 독립적인 상태를 가지는 테스트에 대한 테스트
### 3-1. 개요

#### 3-1.1. 테스트 조합

복수의 독립적인 상태를 가지는 테스트의 경우, 가능한 전체 경우의 수는 각 상태가 가진 경우의 수의 곱으로 결정된다. 이로 인해 테스트의 독립 상태가 늘어날수록 생성해야 하는 테스트가 기하급수적으로 늘어나는 상태 폭발 문제를 마주할 수 있다.

UniTest에서는 각 독립 상태에 대한 단위 테스트를 정의하면 이를 조합하여 모든 가능한 상태에 대한 테스트 케이스를 자동으로 생성하는 방식을 채택하였다.

#### 3-1.2. 테스트 확장

테스트를 조합할 때는, 기존 테스트를 기반으로 다른 테스트를 계층적으로 확장하는 방식으로 동작한다. 예를 들어 기존 테스트 A와 확장된 테스트 B가 있다면 다음과 같은 순서로 테스트를 진행한다.

1. A의 Arrange 실행 후 B의 Arrange 실행
2. A의 Act 실행
3. B의 Assert 실행 후 A의 Assert 실행

이러한 실행 순서를 채택한 이유는 다음과 같습니다.

1. 객체 초기화 순서 보장: 기본 상태(A)를 먼저 초기화한 후에 확장 상태(B)를 초기화해야 올바른 초기화가 이루어질 수 있다.
2. 작업의 단일성 유지: 하나의 테스트 흐름에서는 한 번의 작업만을 수행하는 것이 원칙이므로 기본 상태에 해당하는 A의 Act를 실행한다.
3. 검증 유효성 확보: 확장 부분에서 오류가 발생했을 경우 기본 부분에까지 영향을 줄 수 있기 때문에, 기본 부분의 유효성을 확인하기 전에 확장 부분의 유효성을 먼저 확인한다.

#### 3-1.3. 테스트 유형

기존 테스트 A와 이를 확장한 테스트 B가 있을 때, 테스트 B는 A의 Act가 실행될 것임을 전제로 설계되어야 한다. 그러나 B의 Model이 실행할 Act를 직접 설계할 수 없기 때문에 A에게 어떤 테스트 동작을 실행해야 하는지를 명시적으로 전달해야 한다.

UniTest는 이러한 흐름을 위해 객체가 사용 가능한 테스트 유형을 사전에 정의하며, 테스트 생성기는 이 유형을 인자로 받아 해당하는 테스트를 유연하게 구성할 수 있도록 한다.

테스트 유형은 모두 동등한 계층에 존재할 수도 있지만, 하위 여러 유형을 포함하는 유형이 존재할 수도 있다. 이 경우 테스트 생성기는 다음 두 가지 방식 중 하나를 선택할 수 있다.

- 공통 테스트 생성: 하위 유형만을 기준으로 하고 다른 유형과는 관계없이 동일한 테스트 생성
- 분기 테스트 생성: 다른 유형에 따라 각각 다른 테스트 생성

#### 3-1.4. 테스트 생성기

UniTest에서는 각 독립 상태마다 해당 상태에 대한 테스트를 생성하는 생성기가 존재한다. 각 생성기는 독립적으로 동작하며, 하위 생성기는 상위 생성기를 이어받아 전체적으로 테스트를 구성할 수 있다.

테스트의 생성은 하위 생성기에서 시작하여 상위 생성기로 조립되는 방식으로 이루어진다. 하위 생성기는 입력받은 테스트 유형에 해당하는 테스트를 생성한 후, 동일한 유형을 다음 단계의 상위 생성기에 전달하여 추가 테스트 생성을 요청한다. 이후, 자신이 생성한 테스트와 상위 생성기로부터 반환받은 테스트를 조합하여 하나의 확장된 테스트 세트를 구성하여 반환한다.

테스트 유형에 따라 생성기가 상위 생성기를 호출하지 않고 단독으로 테스트를 생성해야 하는 경우도 발생할 수 있다. 이러한 상황에서는 해당 생성기가 사실상 최상위 생성기 역할을 수행하게 되며, Act 생성을 담당하게 된다. 이러한 구조를 통해, 테스트 유형별로 생성된 테스트 메서드를 사전에 정의해두면 모든 레벨의 생성기가 동일한 방식으로 테스트를 구성할 수 있어 설계의 일관성을 유지할 수 있다.

#### 3-2.1. Model

다수의 독립적인 상태를 가지는 테스트의 테스트에서 Model은 다음과 같은 요소로 구성된다.

<img src="../Resources/01-20260420214502.png" alt="UniTest multi-state model structure diagram" width="351">

Lab은 하나의 독립 상태에 대한 테스트를 실행하는 객체이다.
Lab은 다음과 같은 요소로 구성된다.

<img src="../Resources/01-20260420214513.png" alt="UniTest independent lab structure diagram" width="903">

구성 요소

#### 3-2.2-1. Composite Lab

Composite Lab은 각 독립 상태에 해당하는 Lab들을 묶어 순차적으로 실행하는 객체이다.
Composite Lab은 다음과 같은 요소로 구성된다.

<img src="../Resources/01-20260420214522.png" alt="UniTest composite lab structure diagram" width="902">

구성 요소
- Labs: 각 독립 상태에 해당하는 Lab들의 집합

함수
- Execute: 포함된 모든 Lab에 대해 Arrange → Act → Assert 순서로 테스트 실행
- Extend: 새로운 하위 Lab을 추가하여 확장된 Composite Lab 생성

Composite Lab은 Lab과 함께 공통 인터페이스인 ILab을 구현하며, 기존 테스트에서 Lab이 사용되던 모든 위치에 그대로 교체하여 사용할 수 있다. 이로써 기존 Project 구조와 실행 흐름을 수정하지 않고도 Composite Lab을 기존 테스트에 적용할 수 있다.
