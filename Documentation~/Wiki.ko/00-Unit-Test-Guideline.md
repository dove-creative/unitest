## 1. 유닛 테스트 개요

유닛 테스트는 모듈 단위에서의 오류를 사전에 탐지하여 시스템 안정성을 높이고 디버그를 용이하게 하며, 향후 시스템 확장 시 기존 원칙으로 두고 있다.

하지만 체계적이지 못한 테스트는 시스템의 신뢰성을 보장하는 역할을 제대로 수행하지 못할 수 있고, 지나치게 꼼꼼한 유닛 테스트는 과도한 시간과 비용을 유발하여 개발 효율을 떨어뜨리는 원인이 될 수 있다.

Uni Test는 이러한 문제를 해결하기 위해, 상태-동작 표를 기반으로 테스트 범위와 절차를 명확히 하고 체계적이고 효율적인 유닛 테스트를 수행할 수 있도록 설계되었다. 이 문서에서는 그 설계의 바탕이 되는 유닛 테스트 개념과 방법론을 소개한다.

## 2. 상태 - 동작 표

시스템을 체계적으로 검증하려면 시스템이 어떤 상태에서 어떤 기능을 수행할 수 있는지를 명확하게 파악하는 것이 중요하다.

상태 - 동작 표는 시스템이 가질 수 있는 상태를 행으로, 시스템이 수행할 수 있는 기능을 열로 구성한 표이며, 이를 통해 시스템의 모든 동작 시나리오를 체계적으로 확인할 수 있다.

### 2-1. 상태와 동작의 의미

상태란 현재 시스템이 어떤 동작을 어떻게 처리할지 결정하는 조건의 분류이다.
- 예시: `대기 중`, `실행 중`, `종료됨` 등

>단순히 데이터가 비어 있는지, 특정 필드가 채워져 있는지, 내부 컬렉션에 값이 몇 개 있는지는 그 자체만으로 상태가 아니다. 이런 값들은 보통 테스트 입력, 사전 조건, 경계 조건으로 다룬다.
>- 비예시: `리스트가 비어 있음`, `이름 필드가 null임`, `로그가 3개 있음` 등

동작이란 시스템 외부에서 호출하거나 관찰할 수 있는 의미 있는 행동이다.
- 예시: `시작하기`, `중지하기`, `저장하기`, `내보내기` 등

>단순 helper, 내부 구현 세부 사항, 상태에 따른 결과 차이를 만들지 않는 메서드는 상태 - 동작 표의 동작으로 올리지 않는다.
>- 비예시: `문자열 포맷하기`, `중간 컬렉션 정렬하기` 등

### 2-2. 하나의 상태를 가지는 시스템

하나의 상태를 가지는 시스템에서는 시스템이 가질 수 있는 상태와 수행할 수 있는 기능을 하나의 표로 표현할 수 있다. 설명을 위해 이용자가 탑승할 수 있는 전동 킥보드를 예시로 들겠다.

킥보드는 아래와 같은 세 가지 상태를 가진다.

- Idle: 기본 상태
- Mounted: 이용자가 탑승한 상태
- Disposed: 삭제된 상태

또한, 킥보드는 아래의 기능을 통해 동작하거나 자신의 상태를 바꿀 수 있다.

- Mount: 이용자가 킥보드에 탑승 (면허 소지자만 가능)
- Ride: 킥보드 출발
- Stop: 킥보드 정지
- Dismount: 탑승 종료
- Dispose: 킥보드 객체 삭제

이 킥보드의 상태 - 동작 표를 작성하면 아래와 같다.

| State    | -   | Mount              | <    | <                  | <            | Ride               | Dismount | Dispose                    |
| -------- | --- | ------------------ | ---- | ------------------ | ------------ | ------------------ | -------- | -------------------------- |
|          | -   | Licensed           | Same | Not Licensed       | Null         |                    |          |                            |
|          | -   |                    |      |                    |              |                    |          |                            |
| Idle     | -   | **Mounted**        | X    | _InvalidOperation_ | `<Dismount>` | _InvalidOperation_ | -        | **Disposed**               |
| Mounted  | -   | _InvalidOperation_ | -    | _InvalidOperation_ | ^            | `[Ride]`           | **Idle** | `<Dismount>` → `<Dispose>` |
| Disposed | -   | _ObjectDisposed_   | X    | _ObjectDisposed_   | ^            | _ObjectDisposed_   | -        | -                          |

표 설명

- 상태 (왼쪽 열): 시스템이 가질 수 있는 상태 (Idle, Mounted, Disposed) 표시
	- 상태 열과 내용 열 사이에는 -만 존재하는 열을 넣어 구분한다.
- 기능 (위쪽 행): 킥보드가 실행할 수 있는 기능 표시
	- 기능 행과 내용 행 사이는 한 칸 빈 행을 두어 구분하도록 한다.
- 내용
	- `<`: 왼쪽 칸의 기능과 같은 기능 그룹에 속하는 하위 항목임을 표시한다. 즉 왼쪽 기능 열을 이어 읽는 표시이다.
	- `^`: 위쪽 칸의 동작이나 결과를 그대로 따른다는 표시이다. 같은 내용을 반복해서 적지 않고 위쪽 내용을 이어 읽는다.
- 칸: 각 상태에서 특정 기능을 실행했을 때의 동작 표시
	- 상태 변화: **볼드**
	- 고유 기능: `[대괄호 인라인 블럭]`
	- 동작 실행: `<화살 인라인 블럭>`
	- 예외 발생: _기울임_
	- 변화 없음: -
	- 상태 없음: X

이렇게 상태 - 동작 표를 사용하면 시스템의 가능한 동작 시나리오를 체계적으로 확인할 수 있다.

### 2-3. 다수의 독립된 상태를 가지는 시스템

다수의 독립된 상태를 가지는 시스템의 경우, 시스템이 가질 수 있는 상태 유형의 수가 늘어날수록 시스템이 가질 수 있는 상태의 수가 기하급수적으로 늘어나기 때문에 이를 하나의 표로 표현하는 것이 어려울 수 있다. 이럴 경우 각 상태 유형별로 표를 만들고, 이를 종합하여 기능을 호출할 수 있는 종합 기능 표를 제작하여 해당 상태에 매칭하는 방법을 사용한다.

설명을 위해 전의 킥보드에 아래의 두 가지 상태 유형을 추가한다고 하자.

- Battery: 배터리 잔량 상태
	- Available: 배터리가 충분하여 정상 주행이 가능한 상태
	- Discharged: 배터리가 부족하여 정상 주행이 어려운 상태

- Charge State: 충전 진행 상태
	- Not Charging: 충전 중이 아닌 상태
	- Charging: 킥보드 충전 중

킥보드는 각 상태 유형에서 아래의 기능을 수행할 수 있다.

- Checking: 주행 중 배터리 상태 확인
- Charge: 킥보드 정지 후 충전
- Stop Charging: 킥보드 충전 종료
- Mount (기능 추가): 배터리가 Available 상태에서만 Mount 허용
- Ride (기능 추가): 충전 진행 상태가 Charging 상태에서는 Ride 불가, Ride 후 Checking 시작
- Stop (기능 추가): Stop 전 Checking 종료
- Dispose (기능 추가): 충전 진행 상태가 Charging 상태일 경우 충전을 종료하고 Dispose

이때 킥보드의 전체 상태 - 동작 표는 아래와 같이 작성할 수 있다.

#### 표 설명

- Kickboard 표: 시스템의 기본 상태 유형에 대한 상태-동작 표
- Battery 표: Battery 상태 유형에 대한 상태-동작 표
- Charge State 표: Charge State 상태 유형에 대한 상태-동작 표
- Operations 표: 실제로 사용자가 호출할 수 있는 기능에 대한 표

#### 기능의 Override 표시

(override) 표시가 된 열은 Override된 기능을 나타내며, 상위 기능의 호출을 제어하는 역할을 한다.

예를 들어 Battery 상태 유형이 추가된 킥보드의 경우, 상위 기능인 원본 Mount를 하위 기능인 Battery.Mount가 제어하여 배터리가 Available 상태에서만 원본 Mount를 실행할 수 있도록 한다.

#### Operations 표

다수의 독립된 상태를 가지는 시스템의 경우, 사용자가 완성된 인터페이스를 통해 여러 상태를 동시에 제어하는 경우가 일반적이다. Operations 표를 사용하면 각 상태가 어떤 동작에 영향을 받는지 직관적으로 확인할 수 있다.

표를 읽는 방법은 아래와 같다.

1. 상태 유형별 동작을 하위에서 상위 순으로 확인한다.
2. 하위 상태 유형에 해당 동작이 정의되어 있다면 이를 적용한다.
3. 하위 상태 유형에 해당 동작이 없을 경우 상위 상태 유형의 동작을 적용한다.

예를 들어, 추가 상태 유형이 구현되지 않은 킥보드는 모든 동작이 Kickboard 행으로 연결된다. 반면 Battery와 Charge State 상태 유형이 추가된 킥보드의 경우 Charge State, Battery, Kickboard 순으로 동작을 확인하고, 더 하위 상태 유형에서 재정의되지 않은 동작만 상위 상태 유형에서 확인한다.

#### Kickboard

| Kickboard | -   | Mount              | <    | <                  | <            | Ride               | Dismount | Dispose                    |
| --------- | --- | ------------------ | ---- | ------------------ | ------------ | ------------------ | -------- | -------------------------- |
|           | -   | Licensed           | Same | Not Licensed       | Null         |                    |          |                            |
|           | -   |                    |      |                    |              |                    |          |                            |
| Idle      | -   | **Mounted**        | X    | _InvalidOperation_ | `<Dismount>` | _InvalidOperation_ | -        | **Disposed**               |
| Mounted   | -   | _InvalidOperation_ | -    | _InvalidOperation_ | ^            | `[Ride]`           | **Idle** | `<Dismount>` → `<Dispose>` |
| Disposed  | -   | _ObjectDisposed_   | X    | _ObjectDisposed_   | ^            | _ObjectDisposed_   | -        | -                          |

#### Battery

| Battery    | -   | Check         | <              | Mount (override) | Ride (override)            |
| ---------- | --- | ------------- | -------------- | ---------------- | -------------------------- |
|            | -   | Battery > 10% | Battery <= 10% |                  |                            |
|            | -   |               |                |                  |                            |
| Available  | -   | -             | **Discharged** | `[base]`         | `[base]` → `[Use Battery]` |
| Discharged | -   | **Available** | -              | -                | -                          |

#### Charge State

| Charge State | -   | Charge                      | Do Charge     | Stop Charging    | Mount (override)                          | Ride (override)    | Dispose (override)           |
| ------------ | --- | --------------------------- | ------------- | ---------------- | ----------------------------------------- | ------------------ | ---------------------------- |

## 3. 테스트 원칙

유닛 테스트를 설계할 때는 아래의 원칙을 준수해야 한다.

### 단일 검증 원칙

테스트는 둘 이상의 동작이나 상태 변화를 동시에 검증해서는 안 된다.

이 원칙은 특정 기능의 작동을 방해요소 없이 온전히 검증하여 해당 기능에 대한 테스트의 신뢰성을 높이고 디버깅을 용이하게 하는 역할을 한다.

#### 잘못된 검증 예시

킥보드의 Mount와 Ride 기능을 연달아 수행하여 두 기능을 동시에 검증하는 경우

>Kickboard.Mount() 호출 후 Kickboard.Ride()가 성공적으로 동작하는지 확인

위의 경우 테스트 실패 시 문제가 어느 기능에서 발생했는지 명확히 파악하기 어려우며, 테스트가 성공해도 각 기능이 의도한 대로 작동했는지를 보장할 수 없다.

#### 올바른 검증 예시

1. 우선 Mount를 검증한다.

>Kickboard.Mount() 호출 후, 상태가 Mounted로 변경되었는지 확인

2. 검증된 Mount를 기반으로 Ride를 별도로 검증한다.

>ex) 상태가 Mounted인 상황에서만 Ride() 호출이 성공적으로 동작하는지 확인

이렇게 하면 테스트 실패 시 원인을 명확하게 파악할 수 있으며, 테스트의 성공이 해당 기능이 의도한 대로 작동했음을 보장할 수 있게 된다.

## 4. 테스트 방법

체계적인 유닛 테스트는 시스템의 신뢰성을 확보하는 데 중요한 역할을 한다. 이때 앞서 소개한 상태 - 동작 표를 활용하면 시스템을 체계적으로 검증할 수 있다.

그러나 시스템의 모든 케이스를 테스트하는 것은 많은 시간과 자원을 필요로 하며, 특히 시스템의 규모가 크거나 연속적인 테스트를 진행하는 경우 테스트가 오히려 개발을 방해하는 요소로 작용할 수 있다.

따라서 테스트를 설계할 때는 상황과 목적에 맞는 적절한 테스트 방식을 선택하는 것이 중요하다.이 파트에서는 개발자가 상황에 따라 적합한 테스트 방식을 선택할 수 있도록 다양한 테스트 수행 옵션과 방식을 소개한다.

### 4-1. 테스트 옵션

많은 시스템의 기능은 의도된 상황을 전제로 설계되며, 의도하지 않은 상황에서는 입력을 무시하거나 예외를 던지도록 설계된다.

이러한 상황은 설계상 발생 가능성이 낮은 경우가 대부분이므로, 시간이 부족한 경우 의도하지 않은 상황의 테스트를 생략하는 전략을 선택할 수 있다.

그러나 이는 시스템이 극한 상황에서 어떻게 동작하는지 검증하지 못하는 위험을 초래할 수 있다. 특히, 배포 상태의 코드는 모든 상황에서의 테스트를 수행하여 안정성을 보장하는 것을 원칙으로 한다.

다음은 3가지 테스트 상황에 대한 각 테스트 옵션의 적용 예시이다.

| 상황 | A | B | C |
| --- | --- | --- | --- |
| 킥보드가 Discharged 상태에서 Charge를 호출하는 경우 충전이 진행되는지 확인 | O | O | O |
| 킥보드가 Charging 상태에서 Charge를 호출하는 경우 호출을 무시하는지 확인 | O | O | X |
| 킥보드가 Disposed 상태에서 Charge를 호출하는 경우 에러를 반환하는지 확인 | O | X | X |

#### A. 모든 케이스 테스트

시스템의 모든 가능한 상태에서 기능이 의도한 결과를 반환하는지 확인

- 적용 상황
	- 배포 직전과 같이 시스템의 철저한 안정성이 필요한 경우
	- 극한 상황에서의 동작 검증이 중요한 시스템의 경우 (예: 선형대수 처리 시스템 등)

- 장단점
	- 철저한 검증 가능
	- 자원과 시간이 많이 소모됨

#### B. 예외를 던지지 않는 케이스만 테스트

호출이 허용된 상황에서 정상적인 작동을 수행하는지 확인

- 적용 상황
	- 개발 초기 단계에서 기능의 주요 동작을 빠르게 검증해야 할 때
	- 시간과 자원이 제한된 상황

- 장단점
	- 효율적이고 빠른 검증 가능
	- 예외 상황 검증 부족으로 해당 오류를 사전에 발견하기 어려움

#### C. 의도한 작동에 해당하는 케이스만 테스트

호출이 허용된 상황에서만 작동 결과를 확인

- 적용 상황
	- 프로토타입 단계에서 간단한 동작 검증이 필요한 경우

- 장단점
	- 자원 소모가 적고 빠르게 검증 가능
	- 예외 상황 검증이 완전히 생략되어 신뢰성이 낮음

각 테스트 옵션은 테스트 범위와 목적에 따라 장단점이 다르기 때문에 개발자는 프로젝트 상황에 맞는 적절한 옵션을 선택하는 것이 필요하다. 또한, 필요에 따라 둘 이상의 옵션을 혼합하여 효율성을 높이는 것이 바람직한 선택이 될 수도 있다.

>ex) B 옵션을 바탕으로 테스트하되 Disposed에 대한 동작은 각 상태별로 한 번씩 검증

### 4-2. 테스트 방식

상태–동작 표를 기반으로 테스트 옵션을 적용하여 테스트를 설계하면 시스템을 체계적이고 효과적으로 검증할 수 있다. 그러나 극단적으로 시간이 부족한 경우, 독립된 둘 이상의 상태를 가지는 시스템, 또는 연속적인 기능 테스트가 필요한 경우에는 테스트에 과도한 시간과 자원이 소모될 수 있다.

이 파트에서는 다양한 시스템에 대해 상태–동작 표를 활용한 테스트 설계 방법과 효율적인 테스트 수행 전략을 소개하여 개발자가 상황에 맞는 적절한 테스트 방법을 선택하고 자원을 효과적으로 활용할 수 있도록 돕는다.

#### 0. AI 생성 테스트

AI 생성 테스트는 생성형 AI에게 구현한 시스템의 코드를 입력하고, 프롬프트를 통해 테스트 코드를 작성하도록 요청하여 AI가 독립적으로 테스트에 대한 모든 권한을 가지고 테스트 코드를 생성하도록 하는 방식이다.

이 방식은 다음과 같은 장점을 가진다.

- 단순한 기능이나 개발 초기 단계에서 기본적인 동작을 빠르게 검증할 수 있다.
- 상태-동작 표가 없는 상황에서도 테스트가 가능하기 때문에 개발 시간을 절약할 수 있다.

그러나 AI 생성 테스트에는 다음과 같은 한계도 존재한다.

- AI가 개발자의 의도를 정확히 이해하지 못해, 테스트가 부정확하거나 불완전할 수 있다.
- 복잡한 비즈니스 로직이나 체계적인 테스트가 필요한 경우에는 신뢰성이 떨어질 수 있다.

따라서 AI 생성 테스트는 개발 초기 단계 등 정식 테스트를 수행하기 어려운 상황에서 임시적인 검증 도구로 활용하는 것이 적절하다.

#### 1. 단일 상태 기반 테스트

단일 상태 기반 테스트는 하나의 상태 유형을 가지는 시스템에 대한 테스트 수행 방식이다. 이는 상태-기반 표를 바탕으로 객체가 수행 가능한 동작에 대해, 각 상태에 속한 객체가 의도한 대로 작동하는지를 확인하는 방식으로 이루어진다.

단일 상태 유형을 가지는 킥보드의 A 옵션(시스템의 모든 가능한 상태에서 기능이 의도한 결과를 반환하는지 확인하는 옵션) 테스트를 설계하는 과정은 아래와 같다.

##### 1-1. 상태 검증

테스트를 시작하기에 앞서, 킥보드가 의도한 상태로 전환되는지 확인한다. 이는 단일 검증 원칙을 준수하는 테스트 설계를 위해 필요하다.

1. 킥보드를 생성하고 Idie 상태인지 검증한다.
2. Idle 상태의 킥보드에 Mount 기능을 실행하여 킥보드가 Mount 상태로 전환되는지 확인한다.
3. Idle 상태의 킥보드를 Dispose하여 킥보드가 Disposed 상태로 전환되는지 확인한다.

상태 검증 과정은 후술할 기능 검증 과정과 통합하여 진행하는 것도 가능하다.
이러한 경우, 테스트가 검증되지 않은 기능을 통해 상태를 변경하여 단일 검증 원칙을 위배하지 않도록 유의해야 한다.

##### 1-2. 기능 검증

표의 최상위 행에 작성된 기능을 순서대로 검증한다.
Dismount 기능을 검증하는 케이스는 아래와 같다.

1. Idle 상태의 Dismount 검증
	- Arrange: 킥보드 생성
	- Act: 킥보드의 Dismount 메소드 실행
	- Assert: 킥보드가 여전히 Idle 상태인지 검증

2. Mounted 상태의 Dismount 검증
	- Arrange: 킥보드 생성 후 Mount 메소드를 실행해 킥보드를 Mounted 상태로 전환
			 (상태 검증 테스트에서 Mounted 전환이 정상 작동함을 확인했으므로 이 동작은 단일 검증 원칙에 위배되지 않음)
	- Act: 킥보드의 Dismount 메소드 실행
	- Assert: 킥보드가 Idle 상태로 돌아갔는지 검증

3. Disposed상태의 Dismount 검증
	- Arrange: 킥보드 생성 후 Dispose 메소드를 실행해 킥보드를 Disposed 상태로 전환
	- Act: 킥보드의 Dismount 메소드 실행
	- Assert: 킥보드가 ObjectDisposedException을 반환하는지 검증

##### 1-3. 기타 검증

테스트하지 못한 특수한 경우를 검증한다.
예를 들어 동일 인물이 재탑승하는 경우를 검증할 경우, Mount → Dismount → Mount를 시행하여 기능이 의도한 대로 작동하는지 테스트한다.


한편, 다수의 독립된 상태를 가지는 시스템도 상태-동작 표를 하나로 합친다면 단일 상태를 가지는 시스템과 같은 방법으로 비교적 간단하게 테스트를 설계할 수 있다.

하지만 이러한 방식은 다양한 케이스를 철저하게 테스트하기 어렵고, 상태가 많아질수록 표를 하나로 합치는 작업이 어려워지기 때문에 낮은 레벨의 검증 혹은 임시적인 검증을 수행해야 할 때 적합한 방식이 될 수 있다.

#### 2. 다수의 독립된 상태 기반 테스트

다수의 독립된 상태 기반 테스트는 둘 이상의 상태 유형을 가지는 시스템에 대한 테스트 수행 방식이다. 객체의 상위 상태 유형부터 차례대로 검증하는 방식으로 이루어진다.

다수의 독립된 상태 유형을 가지는 킥보드의 테스트를 설계하는 과정은 아래와 같다.

##### 2-1. 기본 기능 테스트

킥보드를 기본 기능만 구현한 객체로 가정하고 테스트를 진행한다.

##### 2-2. Battery 기능 테스트

킥보드의 Battery 상태 유형 표를 이용하여 관련 기능만에 대한 테스트를 진행한다.
이때 직접 호출할 수 없는 Checking 등과 같은 기능은 Operations 표나 이미 검증된 상위 기능을 이용하는 등의 방법을 통하여 테스트한다.

##### 2-3. 기타 테스트

두 유형을 동시에 사용하는 시나리오 중 앞 단계에서 테스트를 진행하지 못했던 시나리오들을 검증한다. 예를 들어 충전 직후 탑승을 검증할 경우, Stop Charging → Mount를 시행하여 기능이 의도한 대로 작동하는지 테스트한다.

###### 유의사항

상위 기능의 단독 테스트는 하위 기능들이 LSP 원칙을 준수하며 작성되었을 때만 유효한 결과를 얻을 수 있다. 그렇지 않은 경우, 유효한 테스트를 위하여 일부 설정을 조정해야 할 수 있다.

#### 3. 연속 테스트

연속 테스트는 시스템의 기능을 연속적으로 실행했을 때 시스템이 의도한 대로 기능하는지 확인하는 테스트이다. 이는 객체의 상태-동작 표를 기준으로, 처음 주어지는 객체에 대해 해당 객체가 수행할 수 있는 동작을 모두 실행하고, 이후 생성된 각 상태에 대해 동일한 작업을 반복하는 방식으로 이루어진다.

이러한 특징 때문에 연속 테스트는 시스템의 각 기능 유형을 하나씩 검증하는 유닛 테스트보다는 기능간의 복합적인 작용을 전체적으로 검증하는 통합 테스트의 성격을 가진다.

다수의 독립된 상태 유형을 가지는 킥보드의 연속 테스트를 설계하는 과정은 아래와 같다.

##### 3-1. 킥보드 생성

테스트를 진행할 킥보드 객체를 생성한다. 이때 킥보드는 Idle 상태이다.

##### 3-2. 테스트 진행

킥보드를 복제한 후, 각 킥보드에 대해 Operations 표에서 킥보드가 수행할 수 있는 동작을 차례대로 실행한다.

##### 3-3. 결과 확인

각 테스트에서 킥보드가 예상대로 동작했는지 확인한 후, 테스트를 계속 진행할 수 있는 상태의 킥보드에 대해 2번 항목을 반복한다.

##### 연속 테스트의 장점과 한계

###### 장점

연속 테스트는 아주 다양한 시나리오에서 시스템을 철저하게 검증할 수 있어, 시스템의 높은 신뢰성을 확보할 수 있다.

###### 한계

테스트할 경우의 수가 많아지고, 연속 테스트 횟수와 시스템 기능 수가 증가할수록 테스트의 수가 기하급수적으로 늘어난다. 이는 테스트 작성과 실행 시간에 큰 부담을 줄 수 있다.

##### 연속 테스트의 최적화

연속 테스트의 한계를 극복하기 위해 아래의 방법을 사용할 수 있다.

- 테스트 자동화: 테스트를 자동화하여 작성 시간을 줄이고, 멀티 스레드 환경을 활용하여 테스트 실행 시간을 줄인다.
- 테스트 우선순위 지정: 테스트의 중요도에 따라 우선순위를 부여한다. 예를 들어 주요 시나리오를 우선적으로 테스트하고, 덜 중요한 시나리오는 생략하거나 실행 횟수를 줄일 수 있다.
- 극단적인 케이스 생략: 너무 극단적인 시나리오나, 발생 가능성이 극히 낮은 경우는 테스트에서 제외한다.

## 5. 마치며

이 문서가 소개하는 유닛 테스트 가이드라인은 상태–동작 표 기반의 체계적인 유닛 테스트를 통해 시스템의 안정성과 신뢰성을 확보하고, 개발자가 테스트의 설계와 수행을 효과적으로 진행할 수 있도록 하는 것을 목표로 한다.

테스트 옵션, 단일 상태와 다수의 상태 기반 테스트, 연속 테스트 등 다양한 방법론은 각 프로젝트의 상황과 목적에 맞게 조합하여 활용할 수 있으며, 이를 통해 테스트에 소요되는 시간과 자원을 최적화하면서도, 시스템을 신뢰할 수 있는 상태로 유지할 수 있다.

유닛 테스트는 단순히 오류를 탐지하는 도구에 그치지 않는다. 이는 시스템의 신뢰도를 확인하고, 복잡한 기능 간의 상호작용을 검증하며, 미래의 확장을 위한 튼튼한 기반을 제공한다. 필자는 이 가이드라인을 통해 개발자가 보다 체계적이고 효과적인 테스트를 진행할 수 있으리라 기대한다.
